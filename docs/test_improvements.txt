                                                                                                                                 
  2. Fuzzing the checker and compiler                                                                                

  The fuzz targets stop at the parser. A third target that runs the full pipeline (parse → check → compile → run) on
  fuzz input would catch crashes deeper in the stack, especially in the type solver and codegen.

  4. Property-based / round-trip tests

  - Parser round-trip: Parse → pretty-print → re-parse and check AST equality.
  - VM/JIT equivalence: Automatically run the same programs on both backends and assert identical output (the golden
  tests do this, but a programmatic property test could generate random valid programs).

  5. Safety checker coverage

  46 unit tests is good, but the golden tests in arrays/ could be expanded to test more bounds-checking edge cases
  (e.g., aliased array indices, bounds in nested loops, slices of slices).

  6. Regression test workflow

  There's no mechanism to easily add a regression test from a bug report. A script like ./add_test.sh
  tests/cases/regression/issue_42.lyte that scaffolds the file with the right directives would lower the friction.

  7. CI improvements

  - CI doesn't run the benchmarks — even a single-iteration smoke test would catch benchmark regressions.
  - No fuzzing in CI (even a short 30-second run per target would catch regressions).
